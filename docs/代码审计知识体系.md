# 代码审计完整知识体系

> 目的：建立代码审计的完整知识框架，包括审计方向、工具矩阵、AI 增强点、工具链搭建
> 用途：当前审计服务参考 + 未来工具产品化基础

---

## 一、代码审计六大方向

### 方向 1：安全漏洞审计（SAST）

检测可直接被利用的安全漏洞。

| 检查项 | 说明 | 严重度 |
|--------|------|:--:|
| SQL 注入 | 拼接 SQL 而非参数化查询 | P0 |
| XSS | 未转义的用户输入渲染到页面 | P0 |
| 命令注入 | 用户输入拼接到 shell 命令 | P0 |
| 路径遍历 | `../../../etc/passwd` 类攻击 | P0 |
| SSRF | 用户可控的 URL 请求 | P0 |
| 反序列化漏洞 | 不可信数据反序列化 | P0 |
| 认证绕过 | 缺失鉴权中间件、弱密码策略 | P0 |
| 硬编码密钥 | API Key / Token 写在代码里 | P0 |
| 越权访问 | 缺少行级安全/租户隔离 | P1 |
| 加密误用 | MD5/SHA1 做密码哈希、硬编码 IV | P1 |

### 方向 2：依赖供应链审计（SCA）

| 检查项 | 说明 |
|--------|------|
| 已知 CVE 漏洞 | 依赖库是否有公开漏洞 |
| 许可证合规 | 依赖 License 是否与项目兼容 |
| 幻觉包检测 | **AI 特有**：引用了不存在的 npm/pip 包 |
| 过期/废弃包 | 依赖是否已停止维护 |
| 供应链投毒 | 包的来源是否可信 |

### 方向 3：代码质量审计

| 检查项 | 说明 |
|--------|------|
| 代码复杂度 | 圈复杂度 > 15 的方法 |
| 重复代码 | 拷贝粘贴的代码块 |
| 死代码 | 永远不会执行的分支 |
| 命名规范 | 误导性/不清晰的命名 |
| 错误处理 | 空 catch、吞异常、无错误上下文 |
| 代码坏味道 | God Class、长方法、过多参数 |

### 方向 4：性能审计

| 检查项 | 说明 |
|--------|------|
| N+1 查询 | 循环内查询数据库 |
| 内存泄漏 | 未释放的事件监听、闭包引用 |
| 阻塞操作 | 同步 I/O 在主线程 |
| 缓存缺失 | 重复计算/重复请求无缓存 |
| 大对象序列化 | 返回不必要的大数据量 |
| 数据库索引缺失 | 慢查询无索引 |

### 方向 5：架构与设计审计

| 检查项 | 说明 |
|--------|------|
| 循环依赖 | A → B → A 的引用链 |
| 事务边界错误 | 支付/扣款不在同一事务内 |
| 服务粒度 | 微服务太粗或太细 |
| 抽象泄露 | 底层实现细节穿透到上层 |
| 分布式一致性 | 缺少幂等性、补偿机制 |
| 错误传播 | 跨服务错误链路不完整 |

### 方向 6：业务逻辑审计（AI 最难做好的）

| 检查项 | 说明 |
|--------|------|
| 边界条件遗漏 | 负数金额、零库存、过期优惠券 |
| 状态机缺漏 | 订单缺少「已取消但退款中」等中间态 |
| 并发竞态 | 同时扣减库存、重复提交 |
| 业务规则矛盾 | 两个接口对同一规则有不同实现 |
| 权限模型缺陷 | 角色权限不符合实际业务约束 |

## 二、工具矩阵：每个方向用什么工具

### 2.1 安全漏洞（SAST）

| 工具 | 类型 | 语言 | 检测率 | 误报率 | 优点 | 缺点 |
|------|------|------|:--:|:--:|------|------|
| **Semgrep** | 模式匹配 | 30+ | 87% | 42% | 规则好写、扫描快、社区规则多 | 跨文件分析弱、Pro版付费 |
| **CodeQL** | 语义查询 | 15 | 高 | 低 | 最深的数据流分析、误报最少 | 扫描慢（10-60min）、写规则门槛高 |
| **SonarQube** | 综合平台 | 35+ | 50% | 40-60% | 覆盖面广、质量门禁、历史追踪 | 安全问题检出率最低、需自建服务器 |
| **Bearer** | AI增强SAST | 6+ | 中 | 中 | 数据流映射直观、轻量 | 语言支持有限 |
| **Bandit** | Python专用 | Python | 中 | 中 | 免费、简单 | 规则较浅 |
| **Brakeman** | Ruby专用 | Rails | 高 | 中 | Rails 最专业的工具 | 仅 Rails |

**推荐组合：Semgrep（快速门禁）+ CodeQL（深度分析），两个加起来免费覆盖大部分场景。**

### 2.2 依赖供应链（SCA）

| 工具 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **npm audit / pip audit** | 语言自带 | 免费、零配置 | 仅基础 CVE |
| **Snyk** | 商业 SCA | 数据库最全、修复建议好 | 免费版限制扫描次数 |
| **Dependabot** | GitHub 原生 | 自动提 PR 升级 | 仅 GitHub 生态 |
| **Socket.dev** | AI 增强 | 行为分析（不只是 CVE）、检测投毒 | 较新 |
| **OSV-Scanner** | Google 开源 | 免费、多语言、分布式漏洞数据库 | 比商业产品覆盖少 |

**推荐组合：Dependabot（自动）+ OSV-Scanner（CI 门禁），Snyk 按需付费。**

### 2.3 代码质量

| 工具 | 说明 | 针对什么 |
|------|------|---------|
| **ESLint** | JS/TS 标准 | 代码风格、基础规则 |
| **SonarQube** | 综合平台 | 复杂度、重复、测试覆盖率、技术债 |
| **Prettier** | 格式化 | 代码风格统一（不属审计，但算基础设施） |
| **CodeClimate** | 质量平台 | 可维护性评分、churn 分析 |

### 2.4 性能问题

| 工具 | 说明 | 针对什么 |
|------|------|---------|
| **Clinic.js** | Node.js 性能诊断 | CPU/内存/事件循环阻塞 |
| **Lighthouse** | Web 性能 | FCP、LCP、TBT 等 Web 指标 |
| **py-spy** | Python profiling | CPU profiling 无侵入 |
| **pprof** | Go 内置 | 内存/CPU/Goroutine profiling |
| **Database Query Log** | 各框架自带 | 慢查询、N+1 检测 |

**注意：这个方向目前最缺好的自动化工具。很多问题靠人工读代码发现。这是 AI 增强最有价值的空白之一。**

### 2.5 架构分析

**目前没有成熟的开源工具能自动检测架构问题。** 这个方向几乎全靠人工。部分辅助工具：

| 工具 | 作用 |
|------|------|
| **dependency-cruiser** | JS/TS 依赖关系图、循环依赖检测 |
| **Madge** | 可视化模块依赖 |
| **JDepend / ArchUnit** | Java 架构约束 |
| **AI 分析** | 唯一能理解「为什么这样设计不对」的方法 |

### 2.6 业务逻辑

**没有任何工具能自动检测业务逻辑错误。** 这是纯人工/纯 AI 的地盘。AI 能辅助但不可信——必须人最终判断。

## 三、AI 代码特有 Bug 模式（CSA/GT/OWASP 研究）

2025-2026 多项研究（CSA、Georgia Tech、Sonar、Veracode）确认了 AI 生成代码的高频问题：

| 模式 | 频率 | 说明 |
|------|:--:|------|
| **幻觉 API/包名** | 19.7% | 引用不存在的 npm/pip 包 |
| **字符串拼接 SQL** | 高 | AI 默认行为，非参数化查询 |
| **XSS（未转义输出）** | 86% | AI 写的 XSS 相关代码 86% 不安全 |
| **硬编码密钥** | 72% | Android AI 功能 App 中尤其严重 |
| **缺少鉴权中间件** | 高 | AI 生成 API 端点时经常不加 Auth |
| **弱密码哈希** | 中 | MD5/SHA1 用于密码 |
| **缺少输入验证** | 高 | 所有注入类漏洞的根源 |
| **缺失 Rate Limiting** | 高 | AI 默认不写限流 |
| **过度返回数据** | 中 | API 返回整表而非 DTO |
| **缺少并发控制** | 中 | 扣库存/扣余额无乐观锁 |

## 四、工具组装：你的审计工具链

### 4.1 推荐工具链（免费优先）

```
第一层：代码获取
  git clone + 确认版本/branch

第二层：自动扫描（零人工）
  ├── Semgrep OSS          → 安全模式匹配（秒级）
  ├── CodeQL               → 深语义安全分析（分钟级）
  ├── OSV-Scanner          → 已知 CVE 扫描
  ├── ESLint + custom rules → 代码规范
  └── dependency-cruiser   → 循环依赖检查

第三层：AI 辅助分析（你的差异化）
  ├── LLM 逐模块审查      → 发现工具漏掉的问题
  ├── 幻觉包检测           → 检查 package.json/requirements.txt 中是否存在假包
  ├── 业务逻辑理解         → 对比需求文档 vs 代码实现
  └── 架构合理性判断       → 识别 AI 代码特有的设计缺陷

第四层：人工校验（你 15 年经验的核心价值）
  ├── 验证 AI 低自信度的发现
  ├── 过滤各工具的误报
  ├── 补充 AI 100% 漏掉的业务上下文判断
  └── 出具「能不能上线」的最终结论
```

### 4.2 审计一次完整流程的时间预估

| 阶段 | 耗时 | 工具 |
|------|:--:|------|
| 自动扫描 | 5-15 分钟 | Semgrep + CodeQL + OSV |
| AI 分析 | 10-30 分钟 | LLM 逐模块 |
| 人工校验 | 问题数量决定 | 你的经验 |
| 写报告 | 1-2 小时 | 模板化 |
| **总计（<5000 行项目）** | **2-4 小时** | |

## 五、每个工具的部署命令

### Semgrep

```bash
# 安装
pip install semgrep

# 快速扫描（使用社区规则）
semgrep --config=auto .

# 使用 OWASP Top 10 专用规则
semgrep --config=p/owasp-top-ten .

# 仅看严重问题
semgrep --config=auto --severity ERROR .

# 输出 JSON
semgrep --config=auto --json -o semgrep_results.json .
```

### CodeQL

```bash
# GitHub 上直接启用（最简单）
# Settings → Code security → Enable CodeQL

# 本地运行需要 codeql CLI
gh codeql database create my-db --language=javascript
gh codeql database analyze my-db --format=sarif --output=results.sarif
```

### OSV-Scanner

```bash
# 安装
go install github.com/google/osv-scanner/cmd/osv-scanner@latest

# 扫描
osv-scanner --lockfile=package-lock.json
osv-scanner -r .  # 递归扫描所有 lock 文件
```

### dependency-cruiser

```bash
npm install -g dependency-cruiser

# 生成依赖图
depcruise src --output-type dot | dot -T svg > deps.svg

# 检测循环依赖
depcruise src --output-type err --config
```

### 幻觉包检测（AI 特有）

```bash
# 思路：提取 package.json 所有依赖 → 逐个查 npm registry
# 目前没有现成工具，可以自己写脚本
npm view <package-name> 2>&1 | grep -q "404"
```
